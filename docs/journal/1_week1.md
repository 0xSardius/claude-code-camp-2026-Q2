# Week 1 Technical Documentation

## Technical Goal
Build a baseline agent to play tbaMUD with custom architecture that implements all parts of the agent we need need without the use od an SDK.

## Technical Uncertainty
- We don't know whether hand-rolling the agent loop (no SDK, unlike week0's 03_subagent_sdk) will require reinventing meaningful chunks of what an SDK already gives us for free, or whether building it ourselves teaches us something the SDK abstracts away.
- We don't know how much Ruby-vs-Python matters for this kind of agent-harness code, or whether porting step-by-step will surface real semantic gaps between the two languages that a single-language build wouldn't.
- I'm uncertain how well a literal, byte-for-byte port strategy will hold up as the steps get more complex -- past simple config/data-structure steps into the actual agent loop, tool-calling, and real API integration.
- I don't know yet whether keeping Ruby and Python running in parallel (rather than picking one) will be worth the double-maintenance cost, or whether it'll get abandoned once one language proves out.

## Technical Hypothesis
- I hypothesize that building the agent loop by hand, step by step (00_config, 01_struct_skeleton, and more to come), will make it easier to see exactly where an SDK like the Claude Agent SDK is doing real work versus boilerplate, which should make the week0 SDK comparison sharper once week1's loop actually runs a turn.
- I hypothesize that porting Ruby to Python step-by-step, verified by diffing output for byte-for-byte parity, will surface real Ruby/Python semantic gaps (operator differences, truthiness, nil handling) that a single-language rewrite would hide -- and that finding and fixing those gaps is itself a valuable exercise, not just overhead.
- I hypothesize that keeping a written plan (docs/plans/python_port/&lt;step&gt;) before porting each step, with real questions asked and answered rather than defaulted, will keep the two implementations honestly in sync rather than drifting.

## Technical Observations
- Scaffolded week1_baseline/ruby/00_config (Config class, Tasks::Base/Player, a shipped default prompt) as the first hand-built piece -- no SDK, no gems beyond dotenv. Worked cleanly once two path bugs were found and fixed: the bin/00_config launcher's relative path was missing a "ruby/" segment, and examples/example.rb's BOUKENSHA_DIR override was one directory level short of the repo root -- both from the codebase being reorganized into ruby/&lt;step&gt;/ subdirectories after the original paths were written.
- Getting Ruby itself running took more setup than expected: no Ruby was installed on the machine at all, and both `gem install` and `bundle install` initially tried to write into the system gem directory and failed on permissions. Fixed by configuring Bundler to install gems locally per-project (`bundle config set --local path 'vendor/bundle'`) instead of reaching for `sudo` every time -- avoids a real footgun for repeat installs across future steps.
- Ported 00_config to Python (uv + pyproject.toml, package name "boukensha" to match the Ruby module), holding the port to a hard bar: byte-for-byte identical stdout between the Ruby and Python launchers pointed at the same .boukensha/settings.yaml. Hit that bar on the first real attempt after fixing a couple of Python-vs-Ruby formatting differences (true/false vs True/False booleans).
- Repeated the same process for 01_struct_skeleton (Tool/Message/Context, the agent loop's core data structures). Ruby uses Struct.new for Tool/Message (lightweight, readable-for-learning per its own README) but a plain class for Context (real behavior, not just data) -- ported that distinction faithfully into Python as @dataclass vs a plain class, rather than flattening everything to dataclasses for convenience.
- The byte-for-byte parity bar kept paying off: chasing it surfaced a Ruby inclusive-range-vs-Python-slice off-by-one, and Ruby's symbol-keyed hash inspect format ([:direction]) not having a natural Python equivalent (our dict is string-keyed) -- both fixed by hand rather than accepted as "close enough."
- Ran a full multi-angle code review (7 parallel finder agents plus independent verification) on 01_struct_skeleton after it already had passing parity, specifically to check whether "passes the smoke test" and "is actually correct" were the same thing here. They weren't: found 4 real, confirmed bugs the parity test's single code path never exercised -- Context.task was accidentally optional in Python where Ruby requires it, Python's `or` doesn't match Ruby's `||` on falsy-but-valid values like 0 or an empty string, None values rendered as the literal text "None" instead of Ruby's empty-string interpolation, and provider/model validation didn't catch an explicit `false` the way Ruby's does. All four are now fixed, spot-checked with targeted regression tests, and parity still holds.
- Discovered along the way that two independent review finder-agents can disagree with each other (one claimed the falsy/or issue was intentional and fine, another flagged it as a real bug) -- resolved by reading the actual Ruby and Python source directly rather than trusting either agent's conclusion, which is exactly what the verification step is supposed to force.
- Settled .gitignore policy for the two ecosystems as they came up: Gemfile.lock gitignored (Ruby gem/library convention -- consumers resolve their own versions) but uv.lock committed (matches the existing circlemud-world-parser precedent already in this repo) -- a deliberate difference between the two, not an oversight, confirmed explicitly rather than assumed.
- Ported 02_the_registry (Registry#tool/#dispatch). Found and fixed a real Ruby bug while planning the port, not just translating one: Context#register_tool stored a Tool's name as-is, so a Tool built with a symbol name and registered directly (bypassing Registry, which always normalizes to a string) landed under a different key type and became permanently undispatchable even though tool_count still reported it present. Got explicit authorization to fix genuine Ruby bugs in flight rather than port them faithfully, and to backport the same fix to the already-shipped 01_struct_skeleton in both languages once confirmed it was the same latent bug, not a new one. Also confirmed Ruby's `transform_keys(&:to_sym)` (translating string-keyed JSON args to the symbol keys Ruby blocks require) has no Python equivalent to port -- Python's `**kwargs` unpacking is already string-keyed, so the ported dispatch is legitimately shorter, not incomplete.
- Ported 03_prompt_builder (PromptBuilder + five backends: Anthropic, Gemini, Ollama, OllamaCloud, OpenAI). The BOUKENSHA_DIR off-by-one bug pattern generalized: a *new* directory-relative constant (PROMPTS_DIR) shipped with the identical off-by-one on arrival, confirming it's a class of bug (any `File.expand_path(..., __dir__)` math), not a one-off. Found a real interface bug in Ruby itself: PromptBuilder#to_messages called every backend with the same one-argument shape, but three of five backends' to_messages needed two arguments (system+messages) -- an ArgumentError waiting to fire for any caller that used the public delegator directly rather than going through to_api_payload. Fixed by standardizing all five backends on the two-argument shape. Separately discovered Ruby's JSON.pretty_generate and Python's json.dumps disagree on real formatting details (empty-container newline count, non-ASCII escaping) -- needed a hand-rolled pretty-printer to match Ruby's exact output, not just "close enough" JSON.
- Ported 04_api_client (Client -- real HTTP POST with retry/backoff, no third-party HTTP library, matching the Ruby README's explicit "should be visible, not hidden behind a library" stance). This is where the byte-for-byte parity bar that held for four straight steps broke down for a real reason, not a shortcut: a live API call returns a fresh response ID and freshly-generated model text every time, so there's no "correct" byte-diff to chase. Replaced it with an explicit, user-confirmed methodology instead of silently loosening the bar: one real round-trip per language to prove the client actually works, plus one deliberately-invalid-key round-trip (free, deterministic, real network) to verify the error path, plus direct code review for the genuinely deterministic parts (retry counting, payload construction). Code review caught two real bugs the live call itself didn't surface: the response was never closed on the success/error paths (a socket leak, invisible in a single manual run but real in a long-running agent loop), and the body read happened outside the block that catches transient errors, so a connection reset mid-download would crash unretried instead of being wrapped in the client's own error type.
- Ported 05_agent_loop -- the actual tool-calling loop, and the first step big enough to need a real architectural decision: five providers return five different tool-call response shapes, so every backend now implements parse_response (raw -> one common `{stop_reason, content}` shape) and, for everything but Anthropic, a private inverse method to rebuild that provider's own wire format when replaying history. Also created the python-port Claude Code skill this session, codifying the process (orient, research via diff-against-predecessor, decide what needs asking vs. what's a known fix, write a plan doc, execute, verify, review, close out) after running it identically five times by hand. Flagged, rather than silently kept fixing, that the same two bugs (register_tool normalization, the to_messages arity fix) had now regressed in two consecutive steps -- decided with the user to keep reapplying the known fix for now rather than investigate the Ruby-side scaffolding process, but to keep flagging it if it kept happening. It did: cost of running the code-review workflow at "high" effort (~400-500k tokens per step, over 2M tokens across five reviews) became a real, explicitly-raised concern -- resolved by dropping to the review skill's "medium" tier for future steps rather than dropping review entirely, since every single review so far had found a real bug the parity/live testing missed.
- Ported 06_the_logger (structured JSONL session logs, plus a module-level Boukensha singleton for config/debug/quiet state). Surfaced a genuinely new class of Ruby/Python gap, not a repeat of an established one: Ruby evaluates a keyword argument's default *expression* fresh on every call, but Python evaluates a default *value* once, at function-definition time. Agent's Ruby constructor defaults to `logger: Logger.new`; a naive Python port (`logger=Logger()` in the signature) would have constructed exactly one Logger at import time and silently shared it -- one session file -- across every Agent that didn't pass its own explicitly. Fixed with `logger=None` plus a lazy construction inside `__init__`, and verified directly (not just by reasoning about it) that two Agents built without an explicit logger get distinct instances with distinct session ids. Also the first step where the user asked to actually commit generated artifacts (the JSONL session logs themselves, for instructor evaluation) rather than just code -- required a real .gitignore policy decision (carve out `.boukensha/sessions/*.jsonl` specifically, keep settings.yaml and .env ignored), not just "run it and check."

## Technical Conclusions
- A literal, byte-for-byte-parity port strategy is a genuinely effective way to catch real bugs, not just a purity exercise -- every deviation the parity test caught turned out to matter (formatting, truthiness, nil handling), and the deeper code review caught a further set of bugs the parity test's narrow code path couldn't reach on its own. Parity passing and "correct" are related but not identical, and both checks earned their keep.
- Ruby's `||` and Python's `or` looking similar is a real trap -- they diverge exactly on the values (0, "", explicit false) most likely to appear in real config files as deliberate settings rather than accidents. Documented in the root CLAUDE.md now so it doesn't need rediscovering on the next port step.
- Getting a hand-rolled agent loop this far (config + core data structures, in two languages, verified) with zero SDK dependency confirms it's tractable to build the pieces an SDK normally hides -- worth carrying the comparison forward against week0's 03_subagent_sdk once week1's loop actually runs a real turn.
- Keeping a written plan per port step (with real, explicit decisions logged, not silent defaults) kept the Ruby and Python implementations honestly comparable rather than drifting -- each step's plan doc now doubles as the audit trail for why the Python code looks the way it does.
- Local-first tooling defaults (uv's own venv management, Bundler's local vendor/bundle path) avoided a class of sudo-dependent setup friction that would have compounded badly across many future steps -- worth defaulting to local-scoped installs from the start on any future ecosystem this project picks up.
- The Ruby-truthiness-vs-Python-falsy trap wasn't a one-time discovery -- it recurred at every single step from 02 onward, each time in a *load-bearing* spot rather than a cosmetic one: a `tools: []` override that has to actually disable tool declarations, an explicitly-empty `usage: {}` that has to still be processed, a token count of exactly `0` that has to be treated as present. The pattern held up well enough as a checklist item that it stopped being a "gotcha to rediscover" and became a routine check applied to every new `||`/truthy-check translation on sight.
- The byte-for-byte parity bar didn't survive contact with a real network call (04_api_client), but the discipline behind it did -- rather than quietly relaxing to "looks about right," the project replaced it with an equally rigorous, explicitly-negotiated substitute (one live round-trip proving success, one deterministic failure-path check, code review for everything else). The bar changing shape under real constraints, without the rigor itself eroding, is arguably a bigger validation of the process than parity holding perfectly would have been.
- Regressions -- not new bugs, the *same* bugs reappearing -- turned out to be the dominant failure mode once the port moved past the first couple of steps: register_tool's fix alone regressed six separate times across six steps. This says as much about how the Ruby side is being scaffolded between steps as it does about the port itself, and was worth surfacing explicitly to the user rather than treating each recurrence as a fresh, unrelated finding.
- Independent code review kept its hit rate almost suspiciously high -- a real, confirmed, previously-unknown bug on every single step it ran against, from `01_struct_skeleton` through `06_the_logger` (six for six). That consistency is what justified paying for it at all once the token cost became a real, explicitly-raised concern, but it also justified *not* dropping it entirely -- the fix was scaling the review down (medium effort) rather than removing the check.
- Building the tool-calling loop and its logging layer by hand (05, 06) made concrete what an SDK like the Vercel AI SDK or Claude Agent SDK actually buys you: automatic per-provider usage-field normalization, step-level callbacks that collapse what this project spread across `parse_response`/`tool_call`/`tool_result`/`response` logging calls, and (for the Claude Agent SDK specifically, being single-provider) cost computed for free. What this project still had to hand-roll either way -- the actual JSONL-to-disk writing, a maintained pricing table for multi-provider cost estimation -- turned out to be a small, specific slice of the total problem, not the bulk of it.
- A handful of Python-vs-Ruby gaps turned out to be genuinely new categories, not restatements of "truthiness differs" or "no symbol type": Ruby quietly allows one name to mean two different things across a class method and an instance method (no Python equivalent, forced a rename), and Ruby evaluates a keyword default *expression* fresh per call where Python evaluates a default *value* once at definition time (silently shareable mutable state if ported naively). Neither would have been caught by parity testing alone -- the first because the collision is a naming problem invisible in behavior, the second because a single manual run never constructs two instances of the same class back to back.

## Key Takeaway
- Porting an agent loop from Ruby to Python step-by-step, with a written plan before each step and a rigorous acceptance bar after (byte-for-byte parity where possible, an equally deliberate live-verification substitute once real network calls made that impossible) plus an independent code review on every step, turned "does this look right" into "is this actually right" -- six for six on the review catching a real bug, across data structures, an HTTP client, a five-provider tool-calling loop, and structured logging. The recurring finds weren't random: the same Ruby-truthiness trap and the same handful of regressed fixes kept resurfacing, which is itself the strongest evidence that codifying the process (the gotchas list, and eventually the `python-port` skill itself) was worth doing rather than re-deriving the same lessons by hand every step.
